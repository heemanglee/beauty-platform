# Package Structure Guide

이 프로젝트는 최상위 패키지를 역할이 아니라 기능/도메인 기준으로 나눈다.

## 기본 원칙

- 최상위는 `auth`, `user`, `common`처럼 도메인 또는 횡단 관심사 기준으로 유지한다.
- 애플리케이션 전체를 `controller`, `service`, `repository` 최상위 패키지로 나누지 않는다.
- 도메인 내부는 작은 동안 플랫하게 유지하고, 역할이 커질 때만 하위 패키지를 도입한다.

## `common` 예외 규칙

`common`은 여러 도메인에서 공유되는 횡단 관심사를 담기 때문에 역할별 하위 패키지 분리가 기본값이다.

- `common.api`: 전역 API 예외 처리, 공통 응답 모델
- `common.security`: 인증/인가, JWT, Spring Security 설정

## 도메인 내부 분리 기준

도메인 패키지 안에서 아래 신호가 보이면 역할별 하위 패키지 분리를 고려한다.

- 웹 계층, 서비스 계층, 예외, DTO가 서로 다른 변경 세트로 자주 움직인다.
- 파일 수가 늘어 단일 패키지 탐색 비용이 커진다.
- 외부 API 스키마와 내부 모델을 구분할 필요가 있다.
- 테스트도 역할별 경계에 맞춰 읽는 편이 더 자연스럽다.

## 현재 적용 상태

- `auth`는 인증 API, 서비스, DTO, 예외가 함께 커지고 있어 하위 패키지로 분리한다.
  - `auth.controller`
  - `auth.service`
  - `auth.dto`
  - `auth.exception`
- `user`는 아직 엔티티, 리포지토리, 보조 유틸리티 수준이라 플랫 구조를 유지한다.

## 운영 규칙

- 새 도메인을 만들 때는 우선 플랫하게 시작한다.
- 도메인 내부 분리는 필요 신호가 누적될 때만 수행한다.
- 분리 이후에도 import 방향은 도메인 응집도를 해치지 않도록 최소화한다.
